---
title: Creating Your First Contract
description: Step-by-step tutorial for defining a behavioral contract.
---

This tutorial walks through creating a contract for a simple key-value store component.

## Prerequisites

- Lexicon installed (`cargo install --path crates/cli`)
- A Rust project initialized with `lexicon init`

## Step 1: Start the Contract Workflow

```bash
lexicon contract new
```

This launches an interactive workflow. You will be guided through each section of the contract.

## Step 2: Define Identity

When prompted, provide:

- **Contract ID**: `key-value-store` (kebab-case slug, used as the filename)
- **Title**: `Key-Value Store Contract`
- **Scope**: `Defines the behavior of a basic key-value store with get, set, and delete operations`

## Step 3: Define Invariants

Invariants are conditions that must always hold. For a key-value store:

- `inv-001`: "A key set with a value must return that value on get" (severity: required)
- `inv-002`: "After deleting a key, get returns None" (severity: required)

Each invariant has an ID, description, and severity. Required invariants block verification if violated. Advisory invariants contribute to the score but do not block.

## Step 4: Define Required Semantics

These are behaviors the implementation must provide:

- `req-001`: "get(key) returns None for missing keys"
- `req-002`: "set(key, value) overwrites existing values"
- `req-003`: "delete(key) is a no-op for missing keys"

You can add test tags to link these to specific conformance tests (e.g., `["conformance", "basic"]`).

## Step 5: Define Forbidden Semantics

These are behaviors that must never occur:

- `forbid-001`: "Must not panic on missing key lookup"
- `forbid-002`: "Must not leak memory on repeated set/delete cycles"

## Step 6: Define Edge Cases

Describe specific scenarios with expected behavior:

- `edge-001`: Scenario: "Setting the same key twice" / Expected: "The second value overwrites the first"
- `edge-002`: Scenario: "Deleting a key that was never set" / Expected: "No error, no-op"

## Step 7: Review and Accept

The workflow presents the complete contract as TOML for review. You can:

- **Accept** -- Save the contract as-is
- **Refine** -- Provide feedback to adjust
- **Abort** -- Cancel without saving

## Result

The contract is saved to `specs/contracts/key-value-store.toml` with `draft` status and `experimental` stability.

## Step 8: Verify the Contract Exists

```bash
lexicon contract list
```

Output:

```
Contracts
  key-value-store
```

## Step 9: Generate Conformance Tests

```bash
lexicon conformance add key-value-store
```

This generates a test harness at `tests/conformance/key_value_store.rs` with stub test functions for each invariant, required semantic, and forbidden semantic.

## Step 10: Sync to CLAUDE.md

```bash
lexicon sync claude
```

Your contract's invariants, required semantics, and forbidden semantics are now included in the AI context.

## Next Steps

- Fill in the generated conformance test stubs with real assertions
- Run `lexicon verify` to check your implementation against the contract
- Set up [CI integration](/guides/ci-integration/) to verify on every push
- Promote the contract from `draft` to `active` when ready
